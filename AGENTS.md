# AGENTS.md

이 문서는 `/root/fairly/backend`에서 작업하는 AI 코딩 에이전트 전용 지침이다. 사람 개발자 설명문이 아니라, 실제로 코드를 읽고 수정할 때 따라야 할 작업 규칙을 정의한다.

## 1. 프로젝트 개요

- 이 저장소는 `NestJS` 기반 단일 백엔드 프로젝트다.
- 주요 기술 스택은 `TypeScript`, `NestJS`, `Prisma`, `PostgreSQL`, `Redis`, `AWS S3`, `AWS IVS`, `Bootpay`다.
- 엔트리 포인트는 `src/main.ts`, 루트 모듈은 `src/app.module.ts`다.
- API 문서는 서버 실행 시 Swagger 경로 `/api`로 노출된다.
- 도메인별 구조는 대체로 `controller / service / repository / dto / module` 패턴을 따른다.
- 데이터 구조의 기준은 `prisma/schema.prisma`다.

## 2. 기본 작업 원칙

- 요청 범위를 먼저 좁히고, 필요한 파일만 읽는다.
- 변경은 사람이 검토하기 쉬운 단위로 나눈다.
- 한 번에 넓은 리팩터링을 하지 않는다.
- 관련 없는 파일 정리, 네이밍 변경, 포맷 변경은 요청이 없는 한 하지 않는다.
- 기존 프로젝트 패턴을 우선 따른다. 새 추상화나 새 구조는 기존 방식으로 해결이 어려울 때만 도입한다.
- 같은 기능 수정 시 `controller`, `service`, `repository`, `dto`, `prisma schema` 중 영향 범위를 함께 확인한다.

## 3. 탐색 순서

작업 전 아래 순서로 맥락을 잡는다.

1. `package.json`에서 실행 스크립트와 의존성을 확인한다.
2. `src/main.ts`, `src/app.module.ts`에서 부트스트랩과 모듈 구성을 확인한다.
3. 대상 도메인의 `module`, `controller`, `service`, `repository`, `dto`를 확인한다.
4. DB 필드나 relation이 걸린 변경이면 `prisma/schema.prisma`를 먼저 확인한다.
5. 인증, 결제, 방송, 채팅 관련 변경이면 각각 `auth`, `payment`, `ivs`, `redis`, `aws` 연관 모듈까지 확인한다.

## 4. 수정 기준

- 작은 수정으로 해결 가능한 문제를 큰 구조 변경으로 풀지 않는다.
- 기존 함수, DTO, 응답 형식, 예외 처리 패턴을 최대한 유지한다.
- 서비스 계층에 비즈니스 로직을 두고, repository는 데이터 접근 책임에 가깝게 유지한다.
- 새 필드, 새 상태값, 새 에러 케이스를 추가하면 해당 요청/응답 흐름에서 누락된 곳이 없는지 함께 확인한다.
- 여러 변경이 필요하면 한 커밋 또는 한 패치 안에서도 논리적으로 구분되는 단위로 작성한다.
- 사용자가 원하지 않은 대규모 rename, 파일 이동, 레이어 재구성은 하지 않는다.

## 5. DB 및 외부 연동 안전 규칙

이 프로젝트는 실제 서비스 자원과 연결될 가능성이 높다. 아래 규칙을 기본값으로 따른다.

- `prisma/schema.prisma`를 수정할 때는 스키마 영향 범위와 데이터 영향 범위를 먼저 설명한다.
- `prisma migrate`, 대량 데이터 수정, 삭제 쿼리, 운영 DB를 건드릴 수 있는 명령은 사용자가 명시적으로 요청한 경우에만 수행한다.
- 실제 `AWS`, `Bootpay`, `Google OAuth`, `Redis` 연결이 발생하는 실행은 기본적으로 피한다.
- `.env`가 없거나 필수 환경변수가 비어 있으면 임의의 값으로 실행하지 않는다.
- 서버 실행 전 `src/config/environment.validator.ts` 기준의 필수 환경변수를 확인한다.
- 비밀키, 토큰, 실제 사용자 정보, 결제 식별자 등 민감한 값은 출력하거나 문서에 그대로 남기지 않는다.
- 결제, 인증, 본인인증 로직 수정 시에는 어떤 상태값, 요청값, 응답값, 예외 흐름이 바뀌는지 함께 설명한다.

## 6. 검증 규칙

- 이 프로젝트에서는 테스트를 기본 작업으로 간주하지 않는다.
- 요청이 없는 한 테스트 실행을 전제로 작업하지 않는다.
- 기본 검증은 코드 경로 점검, 타입/의존성 영향 검토, 관련 파일 간 정합성 확인 위주로 한다.
- 사용자가 원하면 그때 `build`, `lint`, 테스트 등 추가 검증을 수행한다.
- 검증을 하지 않았으면 하지 않았다고 명확히 말한다.

## 7. 커뮤니케이션 규칙

- 응답은 한국어 중심으로 작성한다.
- 작업 전에는 무엇을 확인할지 짧게 알리고, 작업 중에는 현재 보고 있는 맥락을 간단히 공유한다.
- 추정이 섞인 내용은 사실처럼 단정하지 않는다.
- 선택지가 있는 변경은 바로 밀어붙이기보다, 현재 코드 기준으로 어떤 선택이 자연스러운지 짧게 설명한다.
- 최종 응답에는 변경 결과, 영향 범위, 검증 여부, 남은 리스크를 간단히 정리한다.

## 8. 이 저장소에서 특히 주의할 영역

- 인증: `src/auth`
- 결제 및 후원: `src/payment`, `src/bootpay`, `src/donation`, `src/coin-*`, `src/payout-coin`, `src/settlement`
- 방송/채팅/실시간 상태: `src/ivs`, `src/live`, `src/chat`, `src/redis`, `src/stream`, `src/stream-viewer`
- 사용자 및 관계 데이터: `src/user`, `src/user-detail`, `src/channel`, `src/fan`, `src/fan-level`, `src/bookmark`, `src/blacklist`, `src/manager`

이 영역들은 단일 파일 수정으로 끝나지 않는 경우가 많으므로, 관련 서비스와 DB 필드를 함께 본다.
